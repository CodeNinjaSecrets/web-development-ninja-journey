Developer's most important tool

What is Git?

Management system for Software development.

Git tracks changes, helps developers collaborate, and manage code efficiently, essential for teamwork and showcasing projects on GitHub.



Developers Experience (DevX)

What is the developer experience process? Developer experience (DevEx) refers to the technology, processes and culture that define software development in an organization. It describes how well an engineering culture incentivizes quality work, and how easy it is to get things done.



Git is used for managing code history and collaborating with other developers, enabling efficient version control and team coordination. GitHub is an open-source platform for hosting web applications as projects, facilitating collaborative development, code sharing, and community engagement.

How to use it


For simplicity, we are using bash language, in bash shell.
Make sure we are in the home directory by using command:
cd ~


We use this command to clone (create a copy of) others or our own folder into our local machine from GitHub which has the folder aka repos:
git clone "url_from_github"


We move into that folder by using:
cd repo_name_that_is_saved_in_our_machine


We can then see the folder's content. So, we then use:
code file_name
to create or open a file for writing our code in it.

To add this file into GitHub, we must first add the file to a staging area:
git add file_name


Then create a snapshot of it with:
git commit -m "enter message for snapshot"


Then to push this code finally to GitHub we use:
git push origin main


After making changes to our code we can use:
git status
to see if we have added, committed, and pushed our changes to GitHub.

So far we are working with pushing files to GitHub via Git commands. Now, a scenario where we made a mistake or we want to see our code history, Git comes in clutch. We use:
git log
to see the history of our Git commits aka snapshots, to know the exact code changes in that commit.

Each commit/snapshot contains a hash code. We use:
git show hashcode_of_that_commit
to see the code details of that specific commit.

Now we can choose to reset and go back to our code that was written with our commit as well, this is one of the great reasons for Git. We use these two commands:
git reset --hard hashcode
and then follow it with:
git push origin main --force
for going back to that snapshot.

Now for example you are working on a group project. You have cloned the repos and now your team members have added some code in the repos. Should you clone it again? No just put in the command git pull origin main
We are by default working with the master branch (in other words: default main branch). Developers working in a team, can sometimes dive into another branch within the same project that they are working on, & then later merge it with the master branch.

git checkout branch_name

Branches serve the ability to work independently.


After you add, commit, push changes to your branch in order to grab the changes made in the branch and merge it with your branch you do the command:
git pull origin master

Then finally make a pull request on github UI interface:
(on the web interface)
These output request changes will merge with the master branch correctly with no merge conflicts.

Should you do "git pull origin master" before making a "pull request" ?

Yes!

You should generally do "git pull origin master" before making a pull request.

Why?

Helps ensure that your branch is up-to-date, reduces the risk of merge conflicts.
